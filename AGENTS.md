# Правила для `hookah-db`

## Роль репозитория

Источник истины для схемы PostgreSQL, SQL-миграций, сидов и локального окружения базы данных.

## Стек

- PostgreSQL
- SQL migrations
- seed scripts
- Docker image для локальной БД

## Локальные правила

- Каждое изменение схемы оформляется отдельной миграцией.
- Имя миграции должно отражать предметную область, а не только дату.
- Ограничения, индексы, enum-типы и внешние ключи задаем явно.
- Доменные сущности раскладываем по схемам `auth`, `catalog`, `equipment`, `recipes`, `inventory`, `sales`, `support`.
- Если бизнес-правило можно надежно проверить на уровне БД, предпочитаем ограничение, триггер или тип вместо договоренности в коде.

## Коммиты

- Пишем на русском языке.
- Не смешиваем в одном коммите крупную миграцию и произвольное переименование без причины.
- Перед коммитом проверяем, что порядок миграций читаемый и воспроизводимый.
